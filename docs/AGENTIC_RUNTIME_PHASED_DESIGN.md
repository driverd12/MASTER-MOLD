# Agentic Runtime Expansion: Phased Design

## Status

Draft design for expanding the core template from a durable MCP runtime into a more explicit local agent platform.

## Why This Exists

The current template already has strong runtime primitives:

- durable tasks, runs, locks, decisions, incidents, and TriChat state
- idempotent mutation handling
- policy, preflight, and postflight checkpoints
- roster-driven multi-agent orchestration
- tmux-backed delegated execution

What it does not yet model explicitly is the agentic layer above those primitives:

- goals as durable intents with success criteria
- plans as executable, inspectable structures instead of transient prompts
- artifacts as first-class evidence objects
- domain-pack planners and verifiers as registered runtime capabilities

This design adds those layers without replacing the existing `task.*`, `run.*`, or `trichat.*` surfaces.

## Design Principles

- Preserve local-first posture.
- Reuse current mutation, lock, run, and policy primitives.
- Treat `goal`, `plan`, `task`, `run`, and `artifact` as distinct entities.
- Keep execution reversible and inspectable.
- Favor typed records over transcript-only coordination.
- Make pack extensions register capabilities, not just tools.

## Core Mental Model

```text
Goal -> Plan -> Plan Step -> Task / Run / Tool Execution -> Artifact -> Verification -> Goal Outcome
```

Recommended semantics:

- A `goal` expresses intent and success conditions.
- A `plan` is one candidate strategy to satisfy a goal.
- A `plan step` is an executable or verifiable unit of work.
- A `task` is a durable claimable work item for workers.
- A `run` is the append-only execution ledger for a material operation.
- An `artifact` is durable evidence, output, or evaluation.

## Relationship To Existing Runtime

The new layer should sit above existing runtime objects:

- `goal.*` coordinates intent and acceptance criteria.
- `plan.*` maps intent into structured steps.
- `artifact.*` stores evidence generated by tools, workers, or verifiers.
- `task.*` remains the execution queue.
- `run.*` remains the canonical operation ledger.
- `trichat.*` remains the multi-agent orchestration surface, but can optionally operate against goals and plans.

## Recommended Phases

### Phase 0: Schema And Registry Foundation

Deliverables:

- new storage tables for goals, plans, artifacts, and hook runs
- core TypeScript record types and storage accessors
- pack hook registry added to domain-pack context

Success criteria:

- runtime starts with new schema migrations enabled
- no behavior changes for existing tools
- packs can register planner and verifier hooks, even before they are used

### Phase 1: Goal Namespace

Deliverables:

- `goal.create`, `goal.get`, `goal.list`, `goal.update`, `goal.advance`, `goal.bundle`
- goal lifecycle and event history
- goal-to-task and goal-to-artifact linking

Success criteria:

- a user or autopilot loop can create a durable goal with acceptance criteria
- goal status no longer depends on chat state alone

### Phase 2: Plan Namespace

Deliverables:

- persistent plans with steps and dependencies
- `goal.plan_generate` and `plan.select`
- step readiness and invalidation logic

Success criteria:

- plans survive restarts
- multiple candidate plans can exist for one goal
- selected plan is inspectable and replayable

### Phase 3: Artifact Namespace

Deliverables:

- generic evidence and output records
- artifact linking, supersession, and provenance
- artifacts attached to goals, plans, steps, tasks, runs, and turns

Success criteria:

- execution results and verification outputs are no longer buried in transcript text or JSON blobs
- artifacts can be retrieved and bundled deterministically

### Phase 4: Pack Planner And Verifier Hooks

Deliverables:

- planner hook registry
- verifier hook registry
- hook dispatch tools
- one reference verifier or planner in the CFD pack

Success criteria:

- a pack can add domain-specific planning and verification without modifying core orchestration code
- core runtime can discover and invoke pack hooks generically

### Phase 5: Autopilot And TriChat Integration

Deliverables:

- goal-aware autopilot intake
- plan-to-task expansion
- artifact-first mentorship and verification persistence
- optional goal-driven TriChat turn orchestration

Success criteria:

- autopilot can operate on durable goals instead of heartbeat prompt text alone
- verification and governance are tied to goal outcomes and stored evidence

## Proposed Tool Surface

### `goal.*`

- `goal.create`
  Create a durable goal with acceptance criteria and autonomy settings.
- `goal.get`
  Fetch a goal by id, optionally with related plans, artifacts, and latest runs.
- `goal.list`
  List goals by status, priority, target entity, or source.
- `goal.update`
  Update mutable fields such as title, priority, constraints, budget, or owner.
- `goal.advance`
  Advance goal status with explicit event summary and optional result payload.
- `goal.plan_generate`
  Ask the runtime to generate one or more candidate plans for a goal.
- `goal.bundle`
  Return a deterministic bundle of goal state, selected plan, task links, run links, and artifacts.

### `plan.*`

- `plan.create`
  Persist a candidate plan and its steps for a goal.
- `plan.get`
  Fetch a plan, its steps, and dependency edges.
- `plan.list`
  List plans for a goal or target entity.
- `plan.select`
  Mark one plan as selected for the goal and deselect other candidates.
- `plan.step_update`
  Update a plan step status, executor, metadata, or bound task/run ids.
- `plan.step_ready`
  Evaluate which steps are ready based on dependencies and blocking state.
- `plan.invalidate`
  Mark a plan or plan step invalid with rationale, typically due to changed evidence.

### `artifact.*`

- `artifact.record`
  Persist a durable artifact with links to goal, plan, step, task, run, thread, or turn.
- `artifact.get`
  Fetch one artifact and its link graph.
- `artifact.list`
  List artifacts by scope, type, pack, trust tier, or producer.
- `artifact.link`
  Create a typed relation between artifacts or between an artifact and another entity.
- `artifact.bundle`
  Produce a deterministic bundle for an entity with latest evidence and provenance.
- `artifact.promote`
  Promote an artifact into note or memory space when long-term retrieval value is high.

### `pack.hooks.*`

- `pack.hooks.list`
  List registered planner and verifier hooks, their packs, and supported target types.
- `pack.plan.generate`
  Invoke a specific planner hook or let the runtime choose the best planner for the target.
- `pack.verify.run`
  Invoke a specific verifier hook or let the runtime choose the best verifier for the target.

### Runtime Dispatch Rules

- If `pack_id` and `hook_name` are both provided, dispatch directly.
- If only `hook_name` is provided, resolve uniquely across packs.
- If no hook is provided, select the best hook by `target.entity_type`.
- Hook execution should be logged to `pack_hook_runs`.
- Hook outputs that create evidence should also be persisted through `artifact.record`.

## How This Integrates With Existing Runtime

### Tasks

- plan steps can spawn `task.create`
- step records should store bound `task_id`
- goal bundle can include related tasks

### Runs

- material planner and verifier executions should produce `run.begin -> run.step -> run.end`
- plan step updates should include bound `run_id` where relevant

### TriChat

- `trichat.turn_*` remains unchanged initially
- later, turns may optionally reference `goal_id` and `plan_id`
- decision artifacts from TriChat can map directly into `artifact.record`

### Policy And Verification

- `goal.plan_generate` should respect goal `autonomy_mode`
- `pack.verify.run` complements `postflight.verify`; it does not replace it
- policy outputs and verifier outputs should be recordable as artifacts

## What To Avoid In The First Pass

- do not replace current `task.*` or `run.*` surfaces
- do not require embeddings or vector storage
- do not couple goal state to any single model provider
- do not force all packs to implement planners or verifiers immediately
- do not make autopilot dependent on new namespaces before core CRUD is stable

## Recommended First Reference Usage

Use the CFD pack as the reference implementation for verifier hooks first, planner hooks second.

Why:

- verifier behavior is easier to constrain and test
- CFD already has domain metrics, validations, and reports
- a `cfd` verifier hook can emit artifacts without forcing full goal-driven execution on day one
